# Wireframe Usability Review Checklist

Use this checklist before submitting Lab 05.

| Check Item | Yes/No | Evidence or Screenshot |
|---|---|---|
| The homepage clearly states the problem and main action. | Yes | wireframes/homepage.png — hero box + Search Tutors / Become a Tutor buttons |
| The main user pathway can be followed from start to finish. | Yes | wireframes/homepage.png → record-list.png → detail-view.png → input-form.png; see student_booking_user_flow diagram |
| The input form includes required fields and validation feedback. | Yes | wireframes/input-form.png — required fields flagged, Time field shows error message |
| The record list includes search/filter/category controls. | Yes | wireframes/record-list.png — search box + subject filter dropdown |
| The detail view shows full information and current status. | Yes | wireframes/detail-view.png — photo, bio, subjects, schedule with open/full slot status |
| The admin view includes at least one management action. | Yes | wireframes/admin-view.png — Approve/Reject, Edit/Delete, Update Status actions |
| The dashboard shows meaningful summary information. | Yes | wireframes/dashboard.png — booking counts, upcoming sessions, activity summary |
| Navigation is consistent across screens. | Yes | Same nav bar (Home, Book Session, Find Tutors, Tutor Detail, Dashboard, Admin) on every screenshot |
| The design uses realistic sample data. | No | Wireframes use placeholder labels ([ Name ], [ Subject ]) rather than real sample data — intentional for low-fidelity stage |
| Each screen maps to system-requirements.md. | Yes | homepage.png → FR-01; input-form.png → FR-03, FR-10, FR-11; record-list.png → FR-06; detail-view.png → FR-07, FR-15; dashboard.png → FR-05, FR-08, FR-12; admin-view.png → FR-09. See system-requirements.md traceability matrix. |
