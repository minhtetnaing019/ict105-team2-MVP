# Lab 04 - MVP Feature List

## MVP Decision Rule
The MVP should include the smallest set of features that can demonstrate the core problem-solution fit by the end of the semester.

## MoSCoW Prioritization

| Feature ID | Feature Name | Problem Solved | Related User Story | Priority (Must/Should/Could/Won't) | Technical Complexity (1-5) | User Value (1-5) | Evidence Strength (1-5) | Include in Final Prototype? |
|---|---|---|---|---|---:|---:|---:|---|
| F01 | Tutor Search by Subject | Students rely on informal networks (chat groups, asking friends) that are slow and unreliable | US-01 | Must | 2 | 5 | 5 | Yes |
| F02 | Tutor Availability Display | No structured way to check availability | US-02 | Must | 3 | 5 | 5 | Yes |
| F03 | Session Booking Request | Students settle for tutors they aren't confident in due to slow, informal arrangement | US-03 | Must | 3 | 4 | 5 | Yes |
| F04 | Tutor Profile Listing (subjects + availability) | Tutors are willing but lack visibility | US-04 | Must | 2 | 4 | 5 | Yes |

## Must-Have Features
- Feature 1: Tutor Search by Subject (F01) — directly answers the core evidence that students can't find tutors for specific subjects and fall back on informal networks.
- Feature 2: Tutor Availability Display (F02) — evidence explicitly states "no structured way to check availability."
- Feature 3: Session Booking Request (F03) — addresses the evidence that students settle for tutors they aren't confident in, by replacing informal chat arrangement with a direct request.
- Feature 4: Tutor Profile Listing (F04) — addresses the evidence that "tutors are willing but lack visibility."

## Should-Have Features
None currently evidence-backed at Should level. (Any additions here — e.g. ratings, notifications, messaging — would need to be flagged as team inference rather than direct Lab 03 evidence.)

## Could-Have / Future Features
None currently evidence-backed. Features like exam-period highlighting are suggested by the "2–3 times per semester, concentrated around exams" finding, but no respondent directly requested this feature — team judgment call if included.

## Not in MVP
No additional features are proposed beyond F01–F04, since these are the only ones directly traceable to specific statements in the Lab 03 discovery evidence (informal/unreliable discovery, lack of availability visibility, low confidence in tutor quality, tutor invisibility). Any feature beyond these four — ratings, payments, messaging, AI matching, mobile app — would need to be justified separately as a team assumption, since the current evidence doesn't speak to them.
